# CTO Review - Sprint 1

## Scope
Backup/Restore Integrity — Full-state export, versioned import, and initial-state alignment for Spargn Ayiti.

## Findings

### B8 Fix — availableFunds Initial State (APPROVED)
The original `useState<number>(10500)` was a demo seed that conflicted with `clearAllData()` which resets to `0`. A first-run user saw 10,500 HTG in available funds, while a user who just cleared data saw 0. Aligned on `0` — if a persisted value exists in localStorage, the load effect overwrites it immediately. **No behavior change for returning users**; only first-run and post-reset states are affected, and the alignment is correct.

### B5 Fix — Full-State Export (APPROVED)
The original `exportDataJSON()` exported only 5 fields (`goals`, `completedGoals`, `contributions`, `rates`, `language`), silently dropping every v1.3+ slice. Users relying on backup to migrate devices lost envelopes, profiles, income transactions, vision items, subscriptions, profile data, PIN lock settings, available funds, and the two localStorage-only slices (Sòl, Kòb Sekou). The fix adds all missing fields and a `version: 2` schema marker. The Sòl and Kòb Sekou values are snapshot at the localStorage boundary (Dashboard owns them directly), with proper try/catch on `JSON.parse` for `paidWeeks`.

### B6 Fix — Versioned Import with Migration Guard (APPROVED)
The original `importData()` had no version detection and restored only 5 fields. The fix introduces:
- **v1 (no version field)**: restores only the original 5 fields, preserves existing v2+ state, and warns the user via toast that the restore was partial.
- **v2+**: restores all known fields with proper type guards (`typeof`, `Array.isArray`, `!== undefined` for falsy-safe primitives). Unknown future versions are forward-compatible — known fields are restored, unknown fields are silently ignored.
- **Invalid version type**: rejected with a specific error toast.

### Finding: Double Toast on Version Rejection (LOW)
When `importData()` rejects a backup with an invalid version field, it shows an error toast and returns `false`. The `SettingsPage` handler then shows a second generic error toast. Result: two overlapping error messages for one error event. **Not fixed** — would require modifying Sprint 0 UI code (LOCKED). Impact is minimal; both toasts auto-dismiss after 4 seconds.

### Finding: Missing Shape Validation on Imported Data (MEDIUM)
`JSON.parse` returns `any`, and the import function applies parsed data to state setters without validating object shapes. For example, `if (data.goals) setGoals(data.goals)` would accept a string or object where an array is expected. This same pattern exists in Sprint 0's localStorage load effect (LOCKED). Adding validation only to the import function would create inconsistency. **Not fixed** — risk is mitigated by the fact that backup files are generated by the app itself; only manually-edited or corrupted files could trigger this.

### Finding: clearAllData() Doesn't Clear Sòl/Kòb Sekou localStorage Keys (MEDIUM)
`clearAllData()` removes the main `STORAGE_KEY` but does not clear the 6 Dashboard-owned localStorage keys (`spargn_sol_hand`, `spargn_sol_turn`, `spargn_sol_week`, `spargn_sol_paid`, `spargn_sol_cycle_count`, `spargn_emergency_fund`). After a factory reset, the Dashboard still reads the old Sòl and Kòb Sekou values. **Not fixed** — `clearAllData` is Sprint 0 validated code (LOCKED). This should be addressed in Sprint 2 when Sòl/Kòb Sekou are lifted into AppContext.

### Finding: Comment-Code Discrepancy on Future Version Handling (LOW)
The original patch comment stated "Unknown future versions: refuse with a clear toast and return false." The actual code is forward-compatible — it accepts future versions and restores known fields. **Fixed** — comment updated to accurately describe the forward-compatible behavior, which is the better design choice.
